1. Введение

По мере роста сложности приложения, код необходимо структурировать, чтобы изолировать разные аспекты логики друг от друга Слоистая архитектура - один из самых распространенных способов организации таких систем Она предполагает разделение на уровни (слои), каждый из которых отвечает за строго определенную функцию

2. Определение

Layered Architecture - это архитектурный стиль, при котором система делится на логические слои, взаимодействующие по строгим правилам

Ключевые правила слоя:

1. Имеет собственную зону ответственности
2. Взаимодействует только с соседними (выше- или нижестоящими) слоями
3. Не нарушает архитектурные границы

3. Основная идея

Система разбивается на уровни абстракции Каждый уровень изолирован от деталей реализации других уровней Это позволяет изменять код внутри одного слоя, не ломая другие

4. Классическая структура слоев (сверху вниз)

1. Presentation Layer (Слой представления)
  - Отвечает за: Пользовательский интерфейс (UI), обработку действий пользователя (клики, ввод), отображение данных
  - Что делает: Передает запросы в нижележащие слои
  - Не должен содержать: SQL-запросы, бизнес-логику, сложные вычисления
2. Application / Service Layer (Слой приложения / Сервисный слой)
  - Отвечает за: Реализацию бизнес-сценариев (use cases), координацию действий, управление транзакциями
  - Особенности: Использует Domain Layer для выполнения логики и взаимодействует с Infrastructure Layer через абстракции (интерфейсы)
3. Domain Layer (Слой домена / Бизнес-логики)
  - Отвечает за: Бизнес-сущности, модели данных, доменные ограничения и правила
  - Это сердце приложения
  - Не должен зависеть от: GUI, базы данных, внешних фреймворков
4. Infrastructure Layer (Слой инфраструктуры)
  - Отвечает за: Технические детали: работу с базой данных, сетевые соединения, интеграцию с внешними сервисами (API), отправку email
  - Содержит: Конкретные реализации зависимостей (например, репозитории, работающие с PostgreSQL)

5. Принцип направленности зависимостей (Ключевое правило)

- Направление: Зависимости строятся сверху вниз Верхний слой может использовать нижний, но нижний слой никогда не должен зависеть от верхнего
- Последствия нарушения:
 1. Циклические зависимости
 2. Потеря возможности тестирования
 3. Усложнение поддержки кода

6. Пример структуры проекта (Учебный проект "Менеджер задач")

- Presentation: Веб-интерфейс, формы, таблицы, обработчики кнопок
- Service: Логика создания задачи, проверка данных, обработка исключений
- Domain: Сущности (Задача, Пользователь), бизнес-правила
- Infrastructure: PostgreSQL, репозитории, библиотека psycopg2

Что дает такое разделение?

- Тестировать бизнес-логику (Service/Domain) можно без запуска GUI
- Можно заменить базу данных (например, с PostgreSQL на MySQL), не переписывая Service (нужно только изменить Infrastructure)
- Можно полностью изменить интерфейс (например, с веба на мобильное приложение), не трогая код репозиториев

7. Преимущества и недостатки

Преимущества:

1. Разделение ответственности
2. Повышение читаемости
3. Улучшение тестируемости
4. Легкость замены компонентов
5. Упрощение сопровождения

Недостатки:

1. Избыточная абстракция (иногда код сложнее, чем нужно)
2. Увеличение количества файлов/классов
3. Потенциальное дублирование кода
4. Риск формального подхода (деление ради деления, без понимания)

8. Связь с принципами SOLID

Слоистая архитектура опирается на:

- Single Responsibility (S): Каждый слой отвечает только за свою область
- Dependency Inversion (D): Верхние слои зависят от абстракций нижних слоев, а не от их конкретных реализаций Без соблюдения этих принципов архитектура быстро "деградирует"

9. Когда использовать?

- Проект больше учебного/скриптового масштаба
- Планируется долгосрочное развитие и поддержка
- Предполагается покрытие кода тестами
- Над проектом работает команда разработчиков

Не рекомендуется для небольших скриптов (это усложнит их без необходимости)
